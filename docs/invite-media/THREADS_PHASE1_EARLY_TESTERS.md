# Threads 第一段階（初期テストユーザー呼び込み）— 貼り付け文案

**目的:** 各メディアに **そのまま貼れる本文** をリポジトリで管理する。思考の整理・推敲・他 AI との壁打ちの **履歴**もここに残す（外部メモにだけ置かない）。

**前提**

- **Threads** 1 投稿あたりの文字数上限は **500 文字**（2026 年時点の一般的な制限。変更されたら表と本文を更新）。
- **メイン投稿:** 多くの人が短時間で欲しい **5W1H** を、**一文を短く**、くどくしない。
- **出だし（固定）:** `今、新しいSNSを作ってます。`
- **2 行目以降:** 「既存 SNS の問題／課題への応答」などはここ以降に。
- **コメント欄（スレ内自己リプ）:** もっと詳細を知りたい人・「模倣して先にやる」方向けの補足・リンク用。

## 運用・認証（公開前に読む）

リポジトリや関連ページが **公開**されている場合、**URL を知る人は誰でも**ログイン画面に辿り着ける。

### クライアントだけでは「誰がメール認証しようとしているか」は識別できない

ログアウト状態では、**誰が新規登録・メールログインしようとしているか**をアプリ側だけで判別することはできない（メール＋パスワードを知っていれば、**共有されたダミーアカウントでログイン**されうる）。**画面からメールフォームを消す**と、正当なテストユーザーが **メールでログインできなくなる**ため、**UI を隠すだけの対策は採らない**（2026-04-13 撤回。`docs/DECISIONS.md` 参照）。

### 対策の方向性（検討メモ・未実装が多い）

- **事前に**テスト用のメール＋パスワードユーザを **Supabase Dashboard や signUp で一括登録**し、配布する。**その後**「新規登録だけ止める」は **Auth Hook・プロバイダ設定・招待制**などで別設計（コード大改修なしに近いのは「あらかじめ枠を作って配る」＋サーバ側で新規を拒否、など）。
- **Supabase で Email プロバイダをオフ**にすると、**新規登録だけでなくメールでのログインも**使えなくなる（既存メールユーザの再ログインも不可）。**Google ログインは別**。採用するかは都度判断。

**確定した運用方針**は `docs/DECISIONS.md` に短く残す。ここは備忘と文案専用。
